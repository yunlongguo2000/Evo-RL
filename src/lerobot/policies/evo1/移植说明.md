# Evo1 移植说明

这份说明主要给后续接手同学快速了解：Evo1 是怎么接入到 Evo-RL / LeRobot 训练体系里的，以及这次移植时主要做了什么。

## 1. 总体思路

这次移植没有重写 Evo1 的模型主体，而是沿用了原始 Evo1 已经比较贴近 LeRobot 的那一部分设计：

- policy 本体仍然是 Evo1
- VLM 仍然是 InternVL3
- action head 仍然是 flow matching
- `stage1 / stage2` 两阶段训练逻辑仍然保留

真正的工作重点不在“改模型”，而在于把原始仓库里那套比较集中的训练逻辑，拆开并接到 LeRobot 现有训练基础设施里。

## 2. 为什么训练接入相对自然

原始 Evo1 在推理侧本来就已经有一些 LeRobot 风格的组织方式，比如：

- policy / processor 的概念是比较明确的
- checkpoint 也能整理成标准的模型目录
- 两阶段训练本身就是围绕同一个 policy 展开的

所以这次接入并不是从零抽象一个新 policy，而更像是：

1. 保留 Evo1 模型实现
2. 把原始大训练脚本中的训练逻辑抽出来
3. 接到 LeRobot 的 `make_policy`、`lerobot_train`、processor 保存与加载、dataset 接口、checkpoint 体系里

## 3. 具体做了什么

可以把这次工作理解为三层。

### 模型层

- 在 `src/lerobot/policies/evo1/` 下接入 Evo1 config、model、processor、flow matching action head。
- 保留了原始 Evo1 的关键配置项，比如：
  - `training_stage`
  - `vlm_model_name`
  - `num_layers`
  - `chunk_size`
  - `n_action_steps`
  - `max_action_dim`
  - `max_state_dim`

### 训练层

- 不再使用原始 Evo1 那个集中的训练脚本作为日常训练入口。
- 改为通过 LeRobot 统一训练入口 `lerobot_train` 启动 Evo1。
- 在训练配置层面补了：
  - `stage1`
  - `stage2`
  - `resume_pretrain`

其中：

- `stage1`：冻结 VLM，只训练 action head
- `stage2`：继续训练 VLM + action head
- `resume_pretrain=true`：用于从 stage1 权重切到 stage2，但不恢复 optimizer / scheduler / step

### 工程集成层

- Evo1 被接进了 LeRobot 的 policy registry
- 训练、推理、record 都能通过统一的 `make_policy` / processor 体系加载
- checkpoint 会保存成 LeRobot 标准格式：
  - `config.json`
  - `model.safetensors`
  - `policy_preprocessor.json`
  - `policy_postprocessor.json`

这样后面：

- 训练完成后直接推理
- human-in-loop record 里加载 policy
- stage1 切 stage2

都会更自然，不需要再写很多原始 Evo1 仓库特有的胶水代码。

## 4. 与原始 Evo1 训练实现的关系

要注意，这次移植不是逐行复刻原始训练脚本，而是做了“语义尽量一致、工程上接入 LeRobot”的版本。

换句话说：

- 核心模型和两阶段训练思路是沿用原始 Evo1 的
- 但训练执行框架已经变成 Evo-RL / LeRobot 这套

这意味着：

- 训练配置方式不同
- checkpoint 组织方式不同
- mixed precision / distributed wrapper 的工程边界也会和原始仓库不完全一样

所以如果后续要做“原始 Evo1 vs 接入版 Evo-RL Evo1”的严格对照实验，需要把：

- 数据
- batch size
- 卡数
- precision
- stage 切换方式

都尽量对齐后再看结果。

## 5. 当前建议的理解方式

最适合把现在这版 Evo1 接入看成：

- **不是一个临时脚本适配**
- 而是把 Evo1 真正注册成了 LeRobot 里的一个 policy

这样后续无论是：

- 继续补多 embodiment
- 对齐原始 Evo1 的训练细节
- 修复 processor / tokenizer / checkpoint 可移植性问题

都可以在 LeRobot 统一框架里继续演进，而不用回到原始仓库那种“大而全训练脚本”上反复打补丁。
